# 版本管理

## 目录结构

每个版本对应一个目录，包含以下文件：

```
docs/versions/
├── README.md            ← 本文件：流程说明 + 版本索引
└── vX.Y.Z/
    ├── requirements.md  ← 需求/目标：这个版本要做什么
    ├── design.md        ← 实现方案：怎么做
    ├── progress.md      ← 进度跟踪：做到哪了
    ├── changelog.md     ← 变更记录：做了什么
    └── notes.md         ← 备注：决策记录、踩坑
```

## 文件用途

| 文件 | 什么时候写 | 写什么 |
|------|-----------|--------|
| `requirements.md` | 版本规划阶段 | 版本目标、功能需求、优先级、破坏性变更 |
| `design.md` | 设计阶段 | 架构设计、模块划分、接口设计、技术决策 |
| `progress.md` | 开发过程中持续更新 | 任务状态跟踪（待办/进行中/已完成） |
| `changelog.md` | 发布时整理 | 按 feat/fix/refactor 分类的变更摘要 |
| `notes.md` | 开发过程中随时记录 | 技术决策原因、踩坑记录、待讨论事项 |

## 维护流程

```
1. 规划新版本
   → 创建 versions/vX.Y.Z/requirements.md，列出需求

2. 设计阶段
   → 编写 design.md，记录实现方案和架构决策

3. 开发过程中
   → 随时更新 progress.md，移动任务状态
   → 遇到需要记录的决策或踩坑 → notes.md

4. 发布前
   → 从 git log 整理 changelog.md
   → 更新 progress.md 标记全部完成
   → 确认 requirements.md 中所有需求已覆盖

5. 发布
   → 在 pom.xml 更新版本号
   → 打 git tag
```
